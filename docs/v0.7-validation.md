# v0.7 Validation

v0.7 adds sandboxed validation for approved plans. It may apply approved candidate patches only inside the project sandbox and must not modify production files, cron, systemd, global runtime registries, or `harness/prod`.

## Sandbox-Run Endpoint

```bash
curl -X POST http://localhost:8088/projects/<PROJECT_ID>/approvals/<APPROVAL_ID>/sandbox-run \
  -H "Content-Type: application/json" \
  -d '{"sandbox_mode":"full_sandbox","allow_sandbox_commands":false}'
```

`sandbox_mode` may be `patch_only`, `plan_only`, or `full_sandbox`. The default is `full_sandbox`.

`allow_sandbox_commands` defaults to `false`. When false, commands are validated and logged but not executed.

## Prerequisites

Sandbox validation requires:

- project exists in PostgreSQL
- approval exists and belongs to the project
- approval status is `approved`
- `/srv/ai-lab/projects/<project-id>/approvals/dry-run-validation-result.json` exists
- dry-run validation status is `passed`

## Artifacts

v0.7 writes sandbox artifacts under:

```text
/srv/ai-lab/projects/<project-id>/sandbox/
```

Required files:

```text
/srv/ai-lab/projects/<project-id>/sandbox/sandbox-run-report.md
/srv/ai-lab/projects/<project-id>/sandbox/sandbox-run-result.json
/srv/ai-lab/projects/<project-id>/sandbox/sandbox-command-log.json
/srv/ai-lab/projects/<project-id>/sandbox/sandbox-file-manifest.json
/srv/ai-lab/projects/<project-id>/sandbox/applied.patch
```

Source artifacts are copied into:

```text
/srv/ai-lab/projects/<project-id>/sandbox/input/
```

Expected copied inputs:

```text
modification-plan.md
modification-plan.json
dry-run.patch
dry-run-validation-result.json
patch-validation.json
```

## Patch Sandboxing

Patch validation reuses the v0.6 forbidden path checks. A patch may be applied only when its targets are project-local candidate paths and do not escape `sandbox/workspace/`.

Forbidden targets include:

```text
/etc/*
/usr/*
/var/*
/lib/systemd/*
/etc/systemd/*
/var/spool/cron/*
/srv/ai-lab/harness/prod/*
/srv/ai-lab/runtime/registries/*
```

Patches are never applied directly to production paths.

## Command Restrictions

Blocked command classes include:

```text
sudo *
systemctl *
service *
crontab *
apt *
dnf *
yum *
rm *
mv *
cp * /etc/*
docker compose up *
docker compose down *
docker run *
chmod *
chown *
curl *
wget *
ssh *
scp *
```

Allowed sandbox command categories, only when `allow_sandbox_commands=true`:

```text
pwd
ls
find
cat
grep
python3 -m py_compile
python3 -m compileall
bash -n
git apply --check
```

Commands run without shell expansion, with the sandbox path as `cwd`, bounded stdout/stderr, exit code capture, and a timeout.

## Explicit Non-Goals

v0.7 does not implement live production modification, cron editing, systemd editing, service restart, sudo/root execution, package installation, production harness mutation, global registry mutation, self-improvement, web UI, auth, arbitrary shell execution, Docker sandboxing, Kubernetes, or multi-VM scheduling.

## Ubuntu Validation Steps

Run on Ubuntu Server 24.04 before merge:

```bash
sudo ./install.sh
scripts/healthcheck.sh
curl -X POST http://localhost:8088/projects -H "Content-Type: application/json" -d '{"name":"v0.7","request":"validate sandbox run"}'
curl -X POST http://localhost:8088/projects/<PROJECT_ID>/runtime-inspect -H "Content-Type: application/json" -d '{"target_type":"unknown"}'
curl -X POST http://localhost:8088/projects/<PROJECT_ID>/modification-plan -H "Content-Type: application/json" -d '{"change_request":"validate sandbox run","target_type":"unknown","allow_plan_with_blockers":true}'
curl -X POST http://localhost:8088/projects/<PROJECT_ID>/approvals/<APPROVAL_ID>/status -H "Content-Type: application/json" -d '{"status":"approved","reason":"manual validation","allow_blocked_approval":true}'
curl -X POST http://localhost:8088/projects/<PROJECT_ID>/approvals/<APPROVAL_ID>/dry-run -H "Content-Type: application/json" -d '{"validation_mode":"full_dry_run"}'
curl -X POST http://localhost:8088/projects/<PROJECT_ID>/approvals/<APPROVAL_ID>/sandbox-run -H "Content-Type: application/json" -d '{"sandbox_mode":"full_sandbox","allow_sandbox_commands":false}'
```

Expected results:

- sandbox artifacts are written under the project folder
- `production_modified=false`
- `global_registries_modified=false`
- `harness_modified=false`
- no production files are modified
- no global registry files are modified
- no `harness/prod` files are modified
- no sudo or live production modification occurs
