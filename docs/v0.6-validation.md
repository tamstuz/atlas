# v0.6 Validation

v0.6 adds approved dry-run validation. It validates human-approved modification plans without applying patches, running modifying commands, editing cron, editing systemd, restarting services, using sudo, mutating `harness/prod`, or mutating global runtime registries.

## Approval Transition Endpoint

```bash
curl -X POST http://localhost:8088/projects/<PROJECT_ID>/approvals/<APPROVAL_ID>/status \
  -H "Content-Type: application/json" \
  -d '{"status":"approved","reason":"Reviewed by human.","allow_blocked_approval":false}'
```

Allowed transitions:

- `pending -> approved`
- `pending -> rejected`
- `blocked -> rejected`
- `blocked -> approved` only with `allow_blocked_approval=true`

The endpoint updates PostgreSQL and records audit events. It does not execute anything.

## Dry-Run Validation Endpoint

```bash
curl -X POST http://localhost:8088/projects/<PROJECT_ID>/approvals/<APPROVAL_ID>/dry-run \
  -H "Content-Type: application/json" \
  -d '{"validation_mode":"full_dry_run"}'
```

`validation_mode` may be `patch_only`, `plan_only`, or `full_dry_run`. The default is `full_dry_run`. The approval record must already have status `approved`.

The response includes:

```json
{
  "project_id": "",
  "approval_id": "",
  "status": "passed",
  "validation_report_path": "",
  "patch_validation_path": "",
  "production_modified": false,
  "global_registries_modified": false,
  "harness_modified": false,
  "issues": [],
  "next_step": ""
}
```

## Artifacts

v0.6 writes only these validation artifacts under the project approvals folder:

```text
/srv/ai-lab/projects/<project-id>/approvals/dry-run-validation-report.md
/srv/ai-lab/projects/<project-id>/approvals/dry-run-validation-result.json
/srv/ai-lab/projects/<project-id>/approvals/patch-validation.json
```

PostgreSQL records approval validation metadata and audit events for validation start and completion.

## Patch Validation

The validator reads `/srv/ai-lab/projects/<project-id>/approvals/dry-run.patch` and never applies it.

Blocked targets include:

```text
/etc/*
/usr/lib/systemd/*
/etc/systemd/*
/var/spool/cron/*
/srv/ai-lab/harness/prod/*
/srv/ai-lab/runtime/registries/*
```

Allowed dry-run targets are project-local candidate paths such as:

```text
/srv/ai-lab/projects/<project-id>/workspace/*
/srv/ai-lab/projects/<project-id>/approvals/*
```

Forbidden targets return validation status `blocked`.

## Command Validation

Proposed commands in the modification plan are classified and not executed.

Blocked command classes include:

```text
sudo *
systemctl restart|stop|start|enable|disable *
crontab -e
rm *
mv *
cp * /etc/*
apt install *
apt remove *
docker compose up
docker compose down
```

Each command is classified as `allowed_for_future_review`, `blocked`, or `unknown`, with reasons recorded in the validation result.

## Rollback Validation

The validator checks that the plan includes:

- files affected
- backup strategy
- restore steps
- verification after rollback
- failure trigger
- owner/human approval requirement

Missing items make the dry-run validation `failed` unless another safety issue already makes it `blocked`.

## Explicit Non-Goals

v0.6 does not implement live modification, approval execution, cron editing, systemd editing, service restart, sudo/root execution, package installation, production harness mutation, global registry mutation, self-improvement, web UI, auth, arbitrary shell execution, or multi-VM scheduling.

## Ubuntu Validation Steps

Run on Ubuntu Server 24.04 before merge:

```bash
sudo ./install.sh
scripts/healthcheck.sh
curl -X POST http://localhost:8088/projects -H "Content-Type: application/json" -d '{"name":"v0.6","request":"validate dry-run"}'
curl -X POST http://localhost:8088/projects/<PROJECT_ID>/runtime-inspect -H "Content-Type: application/json" -d '{"target_type":"unknown"}'
curl -X POST http://localhost:8088/projects/<PROJECT_ID>/modification-plan -H "Content-Type: application/json" -d '{"change_request":"validate dry run","target_type":"unknown","allow_plan_with_blockers":true}'
curl -X POST http://localhost:8088/projects/<PROJECT_ID>/approvals/<APPROVAL_ID>/status -H "Content-Type: application/json" -d '{"status":"approved","reason":"manual validation"}'
curl -X POST http://localhost:8088/projects/<PROJECT_ID>/approvals/<APPROVAL_ID>/dry-run -H "Content-Type: application/json" -d '{"validation_mode":"full_dry_run"}'
```

Expected results:

- dry-run validation artifacts are written
- forbidden patches are blocked
- proposed modifying commands are classified but not executed
- no production files are modified
- no global registry files are modified
- no `harness/prod` files are modified
- no sudo or live modification occurs
