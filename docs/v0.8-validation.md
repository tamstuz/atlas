# v0.8 Validation

v0.8 adds production change package generation for human review after sandbox validation passes. It does not execute production changes.

## Endpoint

```text
POST /projects/{project_id}/approvals/{approval_id}/change-package
GET /projects/{project_id}/change-packages
```

Request body:

```json
{
  "change_window": "Sunday 01:00 UTC",
  "operator": "ops",
  "notes": "review only"
}
```

The package endpoint requires:

- approval status `approved`
- dry-run validation result status `passed`
- sandbox validation result status `passed`

## Artifacts

All package artifacts are written under:

```text
/srv/ai-lab/projects/<project-id>/change-package/
```

Required package files:

- `production-change-package.md`
- `production-change-package.json`
- `human-execution-checklist.md`
- `exact-command-plan.md`
- `rollback-checklist.md`
- `preflight-checklist.md`
- `postchange-checklist.md`
- `final-approval-request.json`
- `source-artifact-manifest.json`

Source artifacts are copied into `change-package/source/` when present:

- `modification-plan.md`
- `modification-plan.json`
- `dry-run.patch`
- `dry-run-validation-result.json`
- `patch-validation.json`
- `sandbox-run-result.json`
- `sandbox-run-report.md`
- `sandbox-file-manifest.json`
- `sandbox-command-log.json`

## Human Checklists

The package includes pre-flight, execution, rollback, and post-change checklists. They require humans to confirm the maintenance window, backup or snapshot, current service state, exact target paths, exact command paths, rollback owner, rollback command/path, post-change validation command, executor, approver, and that no autonomous execution is active.

## Command Classification

The exact command plan is for human review only. v0.8 does not call shell runners or command execution services.

Commands are classified with:

- `informational`
- `modifying`
- `privileged`
- `blocked_for_agent`
- `human_only`

Commands containing `sudo`, systemd/service start-stop-restart, cron edits, package install/remove, deletion, moves, copies into `/etc`, Docker execution, `chmod`, or `chown` are marked `blocked_for_agent` and `human_only`.

## Final Approval Record

Package generation creates a final approval record:

```text
approval_type = production_change_package
status = pending
artifact_path = /srv/ai-lab/projects/<project-id>/change-package/production-change-package.md
```

This record means the package awaits human review. It is not execution approval.

## Events

v0.8 records:

- `change_package_started`
- `change_package_sources_loaded`
- `change_package_generated`
- `final_approval_created`
- `change_package_blocked`

## Forbidden Behavior

v0.8 must not execute production commands, apply production patches, edit cron, edit systemd, restart services, run sudo, install packages, delete files, mutate `/srv/ai-lab/harness/prod`, mutate `/srv/ai-lab/runtime/registries`, add a web UI, add arbitrary shell execution, or implement production execution.

## Ubuntu Validation Steps

Do not claim Ubuntu success unless these are run on Ubuntu Server 24.04:

```bash
sudo ./install.sh
scripts/healthcheck.sh
curl -X POST http://localhost:8088/projects -H "Content-Type: application/json" -d '{"name":"v0.8 test","request":"package a reviewed change"}'
curl -X POST http://localhost:8088/projects/<project-id>/runtime-inspect -H "Content-Type: application/json" -d '{"target_type":"unknown"}'
curl -X POST http://localhost:8088/projects/<project-id>/modification-plan -H "Content-Type: application/json" -d '{"change_request":"Review package flow","target_type":"file","allow_plan_with_blockers":true}'
curl -X POST http://localhost:8088/projects/<project-id>/approvals/<approval-id>/status -H "Content-Type: application/json" -d '{"status":"approved","reason":"human reviewed","allow_blocked_approval":true}'
curl -X POST http://localhost:8088/projects/<project-id>/approvals/<approval-id>/dry-run -H "Content-Type: application/json" -d '{"validation_mode":"full_dry_run"}'
curl -X POST http://localhost:8088/projects/<project-id>/approvals/<approval-id>/sandbox-run -H "Content-Type: application/json" -d '{"sandbox_mode":"full_sandbox","allow_sandbox_commands":false}'
curl -X POST http://localhost:8088/projects/<project-id>/approvals/<approval-id>/change-package -H "Content-Type: application/json" -d '{"change_window":"manual validation","operator":"human","notes":"v0.8 validation"}'
curl http://localhost:8088/projects/<project-id>/change-packages
```

Expected results:

- package response status is `packaged`
- `production_modified=false`
- `global_registries_modified=false`
- `harness_modified=false`
- all change-package artifacts exist under the project folder
- final approval record exists with status `pending`
- exact command plan is written but not executed
- no sudo or live production modification occurs
